⬡ Node.js 20+ TS TypeScript 5.3 ⚛ React 18 🗄 SQLite 3 ◈ Tailwind 3.4 ⚡ Vite 5

🚀 Overview

PerfInsight is a full-stack web application built for Performance Testing Centers of Excellence (COE) to upload, organize, compare, and trend-analyze performance test results from any testing tool — JMeter, Gatling, K6, Locust, or custom CSVs.

Instead of manually comparing spreadsheets after every load test run, PerfInsight gives your team a centralized, visual platform to make fast, data-driven decisions.

🔍
Instant Regressions
See transaction-level degradations the moment you upload
📈
Trend Tracking
Multi-metric trends across all test runs over time
🏆
Quality Scoring
Automated 0–100 performance score per comparison
👥
Team Collaboration
Role-based access with per-project allocation
Built for teams that run performance tests regularly and need consistent, automated analysis — not one-off comparisons.

👥 Role-Based Access Control

PerfInsight uses a two-tier role model enforced on both frontend and backend.

🛡 Admin Full Access
✅ Create / edit / delete Projects
✅ Create / edit / delete Sub-Projects
✅ Create / edit / delete Users
✅ Assign projects to users
✅ Upload, compare, view trends (all projects)
✅ Configure transaction filters
✅ Access User Management page
👤 User Restricted
✅ View Projects & Sub-Projects (allocated only)
✅ Upload test results (allocated projects only)
✅ Compare test runs (allocated projects only)
✅ View trend analysis (allocated projects only)
✅ Configure transaction filters
✅ Edit test run names (inline)
✅ Export PDF comparison reports
❌ Cannot create/delete projects or sub-projects
❌ Cannot access User Management
❌ Cannot see unassigned projects

How project access works

  1. An Admin creates a user account via Admin → User Management
  2. During creation, the Admin selects which Projects the user can see
  3. The user immediately sees only those projects across Dashboard, Upload, Compare, and Trends
  4. Sub-projects are implicitly accessible when the parent project is allocated

Last-admin protection

  • The system prevents deleting the last admin account
  • The system prevents demoting the last admin to a regular user
  • A user cannot delete their own account via the API

📖 How to Use the Application

A step-by-step guide covering every feature of PerfInsight — from first login to exporting a PDF report.

1. Login

  1. Open the application URL in your browser.
  2. Enter your Username and Password.
  3. Click Sign In. You will be redirected to the Dashboard.
ℹ️ Sessions last 8 hours. After expiry you will be automatically redirected to the Login page.

2. Managing Projects and Sub-Projects (Admin only)

Creating a Project

  1. Click Projects in the sidebar.
  2. Click New Project and fill in: Project Name, Customer Name, Description (optional).
  3. Click Create.

Creating a Sub-Project

  1. Open a project, then click New Sub-Project.
  2. Enter a Name and optional Description, then click Create.
ℹ️ Deleting a project cascades — all sub-projects, test runs, and uploaded files are removed.

3. Managing Users and Access (Admin only)

  1. Click Admin / User Management in the sidebar.
  2. Click New User and enter Username, Password, and Role (admin or user).
  3. Check the Projects this user can access.
  4. Click Create User.

A user role can only see projects explicitly allocated to them. Admins see all projects.

4. Uploading Test Results

  1. Click Upload in the sidebar (or the Upload New Test Run button on any sub-project page).
  2. Select the Project and Sub-Project.
  3. Drag-and-drop your result file or click to browse. Supported: .csv, .xlsx, .jtl
  4. Fill in optional metadata: Test Run Name, Build / Version, User Load (VUs), Test Tool, Status (Good / Bad / Under Review), Notes.
  5. Click Upload. The parsed transaction count will be displayed.

Renaming a Test Run

Click the pencil icon in the test run row on the Sub-Project Detail page, type the new name, and press Enter or click the save icon.

5. Viewing Test Run Details

  1. Navigate to a sub-project (Projects → Project → Sub-Project).
  2. Click any test run row to open the Transactions popup.

The popup shows every transaction with columns: Transaction Name, Samples, Avg RT (s), P90 RT (s), P95 RT (s), P99 RT (s), Error %, Throughput (req/s), Min (s), Max (s). All response times are shown with 2 decimal places.

Show Configured Transactions Only

If a Transaction Filter is configured for the sub-project, a checkbox "Show configured transactions only" appears at the top of the popup. Checking it limits the view to your configured transaction set. The header updates to show "Transactions (N of Total)".

6. Comparing Test Runs

Starting a comparison

From the Dashboard: expand a sub-project, select Baseline and Compare from the dropdowns, click Compare.
From the Comparison page: select Project → Sub-Project → Baseline → Compare → Run Comparison.

  2. Select a Project, then a Sub-Project.
  3. Select a Transaction from the dropdown (or click a quick-pick button below).
  4. Optionally change the Filter to Good Runs Only.

8. Transaction Filter Configuration

Filters specify which transactions are relevant for a sub-project. Once set, they apply to the Trend Analysis transaction dropdown and the "Show configured transactions only" checkbox in the test run popup.

Opening the filter

Navigate to a sub-project detail page and click Configure Transactions (filter icon button).

Manual selection

  1. Use the search box to filter by name.
  2. Check/uncheck individual transactions.
  3. Use Select All / Deselect All to act on all visible (searched) results.
  4. Click Save.

Import from Excel / CSV

  1. Click Import from Excel inside the filter modal.
  2. Select a .xlsx or .csv file. The importer looks for a column named Transaction, Name, or Label (case-insensitive); falls back to the first column.
  3. All matching transaction names are automatically checked. A confirmation banner shows the import count.
  4. Review the selection and click Save.
💡 You can combine Excel import with manual selection — import a base list, then add or remove individual transactions.

  2. Click Export PDF Report (top-right area).
  3. The PDF is generated in-browser and downloaded automatically.

⚙️ Comparison Engine

Metric Difference Formula

diff% = ((compare_value - baseline_value) / baseline_value) × 100

Regression Detection Thresholds

MetricRegressionImprovementDirection
Avg Response Timediff > +10%diff < −10%Higher = worse
P90 Response Timediff > +10%diff < −10%Higher = worse (primary metric)
Throughput (RPS)diff < −10%diff > +10%Lower = worse
Error Rateabsolute increase > 5%absolute decrease > 5%Higher = worse

Performance Score (0–100)

Score starts at 100

Each regression deducts:
  Critical  (P90 regression > 25%)   → −5 points per occurrence
  Warning   (P90 regression 10–25%)  → −2 points per occurrence
  Error rate regression            → −3 points per occurrence

Comparison Page Layout

When a comparison result is loaded, the page presents results in this order:

  1. Summary at a Glance — 8 stat cards: Compared, Improved, Degraded, Stable, Above 5s, Peak RT, Max Regression, Best Improvement
  2. Executive Summary — Narrative paragraph(s) describing outcome, peak RT, largest regression, best improvement, and a final verdict
  3. Test-Level Metrics Comparison — 6-row table: Total Transactions, Avg Response Time, P90 RT, Error Count, Error Rate, Throughput (Baseline vs Compare with colour-coded change)
  4. Insights — Auto-generated categorized observations
  5. Transaction Comparison (4 tabs):
    • Response Times — Avg RT and P90 RT columns, sortable, color-coded by status
    • Throughput — TPS per transaction with change %
    • Error Rate — Error % per transaction with change %
    • Charts — Avg RT chart, P90 RT chart, Throughput chart, Error Rate chart (all using actual test run names in legends)

PDF Export

The Export PDF Report button (top-right, visible when a comparison result is loaded) generates an A4 landscape PDF entirely in-browser using jspdf + jspdf-autotable. The PDF contains:

  • Navy page header with test run names
  • Comparison overview table (project, build, tool, date)
  • Summary at a Glance cards
  • Executive Summary narrative
  • Test-Level Metrics table with colour-coded change column
  • Insights table
  • Full Transaction Comparison table (page 2+)
  • Timestamped footer with page numbers
ℹ️ PDF generation runs entirely in the browser — no server round-trip required. The filename is automatically derived from the baseline and compare run names.

📡 API Reference

ℹ️ All endpoints except POST /api/auth/login and GET /api/health require Authorization: Bearer <token>

Authentication

# Login
POST  /api/auth/login         # { username, password } → { token, username, role }
GET   /api/auth/me            # → { id, username, role }

# User Management (Admin only)
GET   /api/auth/users         # List all users with project assignments

PUT   /api/auth/users/:id     # Update { password?, role?, project_ids?[] }
DELETE /api/auth/users/:id   # Delete user (cannot delete last admin or self)

Projects

GET    /api/projects                # List (access-filtered for non-admins)
POST   /api/projects                # Create { name, customer_name, description? }  [Admin]
GET    /api/projects/:id            # Get single project with counts
PUT    /api/projects/:id            # Update project metadata  [Admin]
DELETE /api/projects/:id           # Delete + cascade all sub-projects/runs  [Admin]

Sub-Projects

GET    /api/projects/:pid/subprojects           # List sub-projects
POST   /api/projects/:pid/subprojects           # Create { name, description? }  [Admin]
GET    /api/projects/:pid/subprojects/:id       # Get with counts + last run date
PUT    /api/projects/:pid/subprojects/:id       # Update  [Admin]
DELETE /api/projects/:pid/subprojects/:id      # Delete + cascade  [Admin]
GET    /api/projects/:pid/subprojects/:id/filters   # Get transaction filter list
PUT    /api/projects/:pid/subprojects/:id/filters   # Set { filters: string[] }

Test Runs

GET    /api/subprojects/:spid/testruns      # List test runs
POST   /api/subprojects/:spid/testruns      # Upload (multipart/form-data + metadata)
GET    /api/testruns/:id                   # Get single test run
PUT    /api/testruns/:id                   # Update { name?, status?, notes?, build?, user_load? }
DELETE /api/testruns/:id                  # Delete run + uploaded file
GET    /api/testruns/:id/transactions      # Get all transactions for the run

Comparison

🚢 Deployment Guide

Choose a platform below. All guides produce a production-ready, HTTPS-capable deployment.

Recommended for most teams. Two containers (backend + Nginx frontend) connected via Docker network, data persisted with named volumes.

backend/Dockerfile

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
RUN mkdir -p /app/data /app/uploads
ENV NODE_ENV=production
EXPOSE 3001
CMD ["node", "dist/index.js"]

frontend/Dockerfile

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80

frontend/nginx.conf

server {
    listen 80;
    root /usr/share/nginx/html;
    index index.html;

    # SPA routing
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API proxy to backend container
    location /api {
        proxy_pass http://backend:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        client_max_body_size 50M;
    }
}

docker-compose.yml (project root)

version: "3.9"

services:
  backend:
    build: { context: ./backend, dockerfile: Dockerfile }
    container_name: perfinsight-backend
    restart: unless-stopped
    environment:
      PORT: 3001
      DB_PATH: /app/data/perf_comparison.db

🔒 Security Hardening for Production

  • ✅
    Change JWT_SECRET— Set a random 64-char string via openssl rand -hex 32 in your .env or Secret Manager
  • ✅
    Change default admin password— Update in .env or via the Admin UI after first login
  • ✅
    Enable HTTPS— Let's Encrypt (Certbot) or your cloud provider's certificate service
  • ✅
    Set NODE_ENV=production— Disables stack traces in error responses, enables Express optimizations
  • ✅
    Rate-limit /api/auth/login— Add express-rate-limit middleware: 10 attempts per minute per IP
  • ✅
    Restrict CORS origins— Replace wildcard in index.ts with your actual domain (e.g. https://perfinsight.company.com)
  • ✅
    File upload size limit— Review Multer config and set fileSize limit (e.g. 50 MB)
  • ✅
    Backup the SQLite database— Schedule a daily cron to copy perf_comparison.db to S3 / Azure Blob / GCS
  • ✅
    Keep dependencies updated— Run npm audit regularly and apply patches for known vulnerabilities

🐛 Troubleshooting

▶ Backend fails to start on the configured port
# Check for port conflicts
lsof -i :6001

# Verify the data directory exists and is writable
ls -la backend/data/

# View PM2 logs
pm2 logs perfinsight-api
▶ "Invalid or expired token" immediately after server restart

JWT tokens are signed with JWT_SECRET. If the server restarts with a different secret (e.g. the random default), all existing tokens are invalidated. Set a fixed JWT_SECRET value in .env that doesn't change between restarts.

▶ File uploads silently fail or show 413 error
# Ensure uploads directory exists and is writable
mkdir -p backend/uploads && chmod 755 backend/uploads

# Add to Nginx server block to increase body limit:
client_max_body_size 50M;
▶ SQLite "database is locked" error in production

SQLite supports only one writer at a time. In containerized or multi-process deployments, ensure only one backend process is running. WAL mode (already enabled) helps, but concurrent writes from multiple processes will still fail. If you need horizontal scaling, migrate to PostgreSQL using pg + knex or Prisma.

▶ Frontend shows blank page after Nginx/static deploy

The React SPA uses client-side routing. Without try_files $uri $uri/ /index.html; in your Nginx config, any URL other than / will return a 404 on page refresh. Ensure this directive is in your location / block.

▶ Docker containers can't reach each other / API calls fail

Both containers must be on the same Docker network (defined in docker-compose.yml). The Nginx config inside the frontend container must proxy using the backend's service name: proxy_pass http://backend:3001; — not localhost.

▶ Regular user can still see all projects

The user's JWT token contains their role. If you changed a user's role in the database without them re-logging in, they still have an old token with the old role embedded. Ask the user to log out and log back in — the new token will reflect their updated role and project access.
